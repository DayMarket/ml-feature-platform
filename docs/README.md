# ML Feature Platform

`ml-feature-platform` - репозиторий для PySpark-пайплайнов, которые собирают признаки для продуктовых сценариев: поиск, рекомендации, открытие ПВЗ, матчинг, ранжирование и другие ML-задачи.

Платформа делает три основные вещи:

- собирает промежуточные и переиспользуемые таблицы в слое `silver`;
- собирает финальные модельные признаки в слое `gold`;
- при необходимости публикует `gold`-признаки в сервис ранжирования через `upload/ranking_features/v1`.

Важное текущее ограничение: источники для пайплайнов должны быть доступны как Iceberg-таблицы. Если нужных данных еще нет в Feature Platform, источник подбирается среди Iceberg-таблиц через Trino или ClickHouse, а не подключается как произвольный внешний файл или API.

## Где что лежит

- `layers/silver/*/v1` - переиспользуемые агрегаты и промежуточные таблицы.
- `layers/gold/*/v1` - финальные признаки для моделей или downstream-сервисов.
- `upload/ranking_features/v1` - загрузка feature groups в Kafka-топик сервиса ранжирования.
- `scripts/` - CI-синхронизация миграций, dbt sources, Iceberg maintenance и валидации.
- `ci_test/` - локальные regression-тесты для CI-логики.
- `docs/feature_platform_guide.md` - подробная инструкция по созданию, изменению, удалению и публикации признаков.

## Где лежит контракт таблицы

Для любой таблицы источником правды являются файлы ее слоя:

- `config.yaml` - имя таблицы, слой, primary key, owner metadata;
- `migrations/*.sql` - DDL, список колонок и комментарии;
- `job/*.py` - источники, join keys, фильтры, формулы, окна и запись;
- `dag.py` - расписание, task name и зависимости;
- `README.md` - человеческое описание контракта таблицы.

Если нужен актуальный список признаков или ответ на вопрос “есть ли уже такая фича”, попросите агента проверить репозиторий. Общий каталог вручную не поддерживается, потому что он быстро устаревает; актуальное состояние берется из файлов слоя.

## Коротко про слои

`silver` нужен, когда результат может использоваться несколькими признаками или является нормализованным/агрегированным представлением источника.

`gold` нужен, когда таблица уже является финальным feature contract: у нее понятный entity grain, стабильные колонки признаков и она может быть использована моделью или загружена в сервис ранжирования.

## Если нужного источника нет в Feature Platform

Если нужные данные уже есть в `layers/silver` или `layers/gold`, лучше строить новую фичу от этих таблиц: там уже есть контракт, DQ и lineage.

Если нужных данных в Feature Platform нет, источник выбирается среди Iceberg-таблиц. Для этого агент может использовать Trino или ClickHouse: посмотреть схему, доступные поля, значения бизнес-кодов, партиции и свежесть данных. После этого в Feature Platform создается слой, который трансформирует выбранный Iceberg-источник в нужный `silver` или `gold` contract.

## Основные проверки

Перед завершением изменений агент обычно проверяет несколько вещей:

- что layer-конфиги, миграции и базовые repository contracts читаются корректно;
- что синхронизация dbt sources сможет создать или обновить DQ-описания для новых таблиц;
- что синхронизация Iceberg maintenance добавит только таблицы, созданные Feature Platform, и не затронет внешние источники;
- что ranking upload config ссылается на существующие `gold`-таблицы и реальные колонки из migrations;
- что в измененных файлах нет лишних пробелов и других whitespace-проблем.

Локальная проверка Spark-миграций может требовать PySpark и инфраструктурные credentials. В CI миграции проверяются в Spark runtime.
